# 从这里开始 👋

这个文件夹包含安装GuiZang 需要的**全部文件**。请按下面顺序看：

1. **先读 → 本文档“安装手册”章节**：保姆级步骤，照着做即可（每步有“成功标志”）。

## 这个文件夹里有什么

| 名称 | 作用 | 安装人员要不要动 |
| --- | --- | --- |
| `README.md` | 当前项目入口、快速启动和结构说明 | ✅ 先读它 |
| `项目文档汇总.md` | 安装手册、命令速查、小白手册合集 | ✅ 交付查阅 |

| `bff/` `frontend/` | 系统程序（部署时自动用到） | ❌ 不用管，别删 |

## 三句话流程

1. 把整个文件夹拷到**服务器**，进入 `server` 目录运行唯一入口 `./deploy.sh`（云服务器加公网IP）。它会自动完成环境配置、GuizangAI/Ollama 准备、服务端启动、下载页生成和本机 Agent 接入。
2. 部署完屏幕会给出**下载页地址**，在每台员工电脑浏览器打开它，按系统（Windows / Mac / Linux）粘贴一条命令装 Agent。
3. 打开**仪表盘地址**，用默认账号密码 `testadmin` / `testpass` 登录（登录后请尽快改密），在“设备资产”看到电脑“在线”即成功。

> 详细每一步看本文档“安装手册”章节。遇到问题看手册末尾的“出问题对照表”。

## 2. 安装手册.md

# GuiZang · 现场安装手册（给安装人员）

本手册按顺序操作即可，不需要任何编程基础。每一步都写清楚：**敲什么命令、会看到什么、怎么算成功**。

整体只有三件事：
1. 在【服务器】上装好系统（做一次）
2. 在每台【员工电脑】上装 Agent（Windows / Mac）
3. 打开网页确认电脑都上线了

---

## 开始前的准备清单

请先确认拿到这些信息（问客户 IT 或负责人）：

- [ ] 一台**服务器**（Linux 系统），能用账号密码登录（SSH）。
- [ ] 服务器的 **IP 地址**。
   - 公司内网服务器：用内网 IP（如 `192.168.x.x`）。
   - 云服务器（阿里云/AWS 等）：用**公网 IP 或域名**。
- [ ] 服务器能**联网**（要下载软件）。
- [ ] `sentinel` 这个文件夹（U 盘或网盘拷贝过来）。
- [ ] 员工电脑能通过网络访问到服务器。

> 记一个词：**“服务器 IP”**。后面员工电脑装 Agent 全靠它，本手册会教你怎么自动拿到它。

# 第一部分：在服务器上安装（做一次）

## 步骤 1：登录服务器

在你自己的电脑上打开“终端”（Mac/Linux）或“PowerShell”（Windows），输入（把 `用户名` 和 `服务器IP` 换成真实的）：

```bash
ssh 用户名@服务器IP
```

第一次会问 `yes/no`，输入 `yes` 回车；再输入服务器密码（输密码时屏幕不显示，是正常的）。

✅ **成功标志**：命令行开头变成了服务器的名字。

## 步骤 2：把 sentinel 文件夹放到服务器

如果文件夹还在你自己电脑上，**新开一个终端窗口**（不要关上面那个），执行：

```bash
scp -r sentinel/ 用户名@服务器IP:/opt/
```

输入密码，等待传输完成。

> 如果用 U 盘直接插在服务器上，把 `sentinel` 文件夹复制到 `/opt/` 目录即可，跳过这步。

## 步骤 3：进入目录

回到“已登录服务器”的那个终端窗口，执行：

```bash
cd /opt/sentinel/server
chmod +x deploy.sh
```

（`chmod` 这步没有任何输出是正常的。）

## 步骤 4：运行一键安装

**内网服务器**，直接：

```bash
./deploy.sh
```

**云服务器**，必须带上公网 IP 或域名：

```bash
./deploy.sh 你的公网IP
```

## 步骤 5：按提示回答两个问题

安装过程中会问你：

1. `现在就用默认配置继续部署吗？[y/N]`
   → 测试就输 `y` 回车。
2. `是否启用 Agent 注册密码？[y/N]`
   → 建议输 `y`，再回车（留空会自动生成一个密码）。
   → **把屏幕上显示的密码记下来！** 这个密码**不会写进下载脚本**（避免公开泄露）；之后在每台电脑装 Agent 时，脚本会**提示你手动输入**这个密码（见第二部分）。把它私下发给装机的同事即可。

然后脚本会自动下载并启动所有程序。
> 部署完成后脚本会尝试自动打开**仪表盘**（不会自动打开 Agent 安装/下载页）。如需完全禁止自动打开浏览器，可执行 `GUIZANG_NO_OPEN_BROWSER=1 ./deploy.sh`。
**第一次比较慢，可能要 3–10 分钟，请耐心等待，不要关窗口。**

## 步骤 6：看到这个就成功了

安装结束后，屏幕会出现一段框起来的信息，类似：

```
==================================================================
  [OK] 部署完成！
==================================================================

   ★★★  Agent 该填的 IP 就是： 192.168.31.50  ★★★

   老板仪表盘：    http://192.168.31.50:8080
   ★ 在其它电脑上装 Agent：用浏览器打开下面这个【下载页】…
       下载页： http://192.168.31.50:8080/download/
```

✅ **成功标志**：看到 `[OK] 部署完成！`，并记下这两个地址：
- **仪表盘地址**：`http://服务器IP:8080`
- **下载页地址**：`http://服务器IP:8080/download/`

> 提示：刚装完，网页可能要再等 1–2 分钟才能完全打开（后台还在启动）。

### （仅云服务器）步骤 6.5：开放端口

在云服务商控制台找到“安全组”，放行这些**入方向**端口：`1514`、`1515`、`8080`。
不开的话员工电脑连不上。内网服务器忽略这步。

# 第二部分：在员工电脑上装 Agent

> 每台电脑只需装一次。装完会**开机自动运行**，不用管它。
>
> 打开下载页 `http://服务器IP:8080/download/`，有两种方式：
> - **双击安装（推荐给不熟电脑的同事）**：下载 Windows 的 `.exe` 或 Mac 的 `.pkg`，双击运行。
>   （需提前用 `installers/` 里的脚本构建好安装包；没构建则只显示命令行方式。）
> - **命令行安装**：复制页面上的一条命令运行（下面详细说这种）。

## 如果是 Windows 电脑

1. 点开始菜单，输入 `PowerShell`。
2. 在 “Windows PowerShell” 上**点右键 → 以管理员身份运行**（这一步很重要！）。
3. 在弹出的蓝色窗口里，**粘贴下面这一整行**（把 `服务器IP` 换成第一部分记下的 IP），回车：

```powershell
powershell -ExecutionPolicy Bypass -Command "iwr http://服务器IP:8080/download/install-windows.ps1 -OutFile $env:TEMP\install-agent.ps1; & $env:TEMP\install-agent.ps1"
```

4. 跑到一半会出现一行提示 `如有【服务器注册密码】请输入`：
   - 服务器**启用了注册密码** → 把管理员给你的密码**粘贴进去回车**（粘贴用右键或 Ctrl+V）。
   - **没启用密码** → **直接回车**跳过。
5. 等它自动跑完。

✅ **成功标志**：出现绿色字 `[OK] 安装完成，WazuhSvc 正在运行`。

❌ 如果报错提示权限/服务名无效 → 多半是没“以管理员身份运行”，重做第 2 步。

## 如果是 Mac 电脑

1. 打开“启动台”或“访达 → 应用程序 → 实用工具”，找到并打开 **“终端”**。
2. **粘贴下面这一整行**（把 `服务器IP` 换成真实 IP），回车：

```bash
curl -fsSL http://服务器IP:8080/download/install-macos.command -o /tmp/install-agent.command && sudo bash /tmp/install-agent.command
```

3. 会要求输入这台 Mac 的**开机密码**（输入时不显示，正常），回车。
4. 再出现一行提示 `请输入【服务器注册密码】（没设密码就直接回车跳过）`：
   - 服务器**启用了注册密码** → 把管理员给你的密码**粘贴回车**。
   - **没启用密码** → **直接回车**跳过。

✅ **成功标志**：出现 `[OK] 安装完成，Agent 已运行并设为开机自启`。

## 如果是 Linux 电脑/服务器

1. 打开终端（或 SSH 登录到该 Linux）。
2. **粘贴下面这一整行**（把 `服务器IP` 换成真实 IP），回车：

```bash
curl -fsSL http://服务器IP:8080/download/install-linux.sh -o /tmp/install-agent.sh && sudo bash /tmp/install-agent.sh
```

3. 出现提示 `请输入【服务器注册密码】（没设密码就直接回车跳过）` 时：启用了密码就**粘贴回车**，没启用就**直接回车**。
4. 脚本会**自动识别系统**（Ubuntu/Debian/CentOS/RHEL/SUSE）并安装对应版本的 Agent。

✅ **成功标志**：出现 `[OK] 安装完成，GuiZang 监控已运行并设为开机自启`。

> 说明：Agent 版本会**自动对齐服务器版本**（装不上时也可加 `WAZUH_VERSION=4.14.5` 指定）。

# 第三部分：确认所有电脑都上线

在你自己的电脑（或服务器）浏览器里打开**仪表盘地址**：

```
http://服务器IP:8080
```

首次会出现**登录页**，默认账号和密码为 `testadmin` / `testpass`：

- 账号：`testadmin`
- 密码：`testpass`

> 🔐 登录后请尽快点右上角用户名 → **“修改账号密码”**，用原密码验证后改成你自己的（新密码至少 6 位）。

登录后点左边的 **“资产清点 → 设备清单”**，应该能看到刚装过的电脑，状态是**“在线”**（绿色）。

仪表盘左侧共有 5 个页面：
- **总览大屏**：风险分、AI 安全研判、趋势；右上角 **“导出报告”** 可一键生成可打印的安全周报（浏览器里选“打印 → 存为 PDF”即可发给老板）。
- **安全告警**：告警明细（按等级/设备/关键词筛选）、文件变更(FIM)、自动响应记录。
- **漏洞与补丁**：每台设备的 CVE 漏洞与 CVSS 评分，指导打补丁。
- **资产清点**：设备清单、开放端口（红色为高风险端口）、已装软件清单。
- **合规加固**：各设备 CIS 基线评分与 AI 加固建议。

✅ **全部完成！** 装了几台就该看到几台在线。

> 刚装好的电脑可能要等 1–2 分钟才显示在线，稍等刷新即可。

| 现象 | 怎么办 |

> 仍解决不了，把屏幕上的报错信息拍照/复制，发给技术负责人。

## B. Windows 电脑装 Agent

> 必须「**以管理员身份运行 PowerShell**」：开始菜单搜 PowerShell → 右键 → 以管理员身份运行。

### 方式①：一条命令在线安装（推荐）

```powershell
powershell -ExecutionPolicy Bypass -Command "iwr http://<服务器IP>:8080/download/install-windows.ps1 -OutFile $env:TEMP\install-agent.ps1; & $env:TEMP\install-agent.ps1"
```

> **注册密码怎么输：** 服务器启用了注册密码时，脚本运行中会停下并显示 `如有【服务器注册密码】请输入`，把管理员私下给你的密码**粘贴进去回车**即可；没启用密码就**直接回车**跳过。（密码不再写进脚本，更安全）

### 方式②：已下载脚本到本地（如 C:\install）

```powershell
cd C:\install
powershell -ExecutionPolicy Bypass -File .\install-windows.ps1
```

### 方式③：手动指定参数（用未注入信息的原版脚本时）

```powershell
powershell -ExecutionPolicy Bypass -File .\install-windows.ps1 -Manager <服务器IP> -RegistrationPassword "注册密码"
```

### Windows 验证与服务控制

```powershell
Get-Service WazuhSvc                                            # 状态应为 Running
Get-Content "C:\Program Files (x86)\ossec-agent\ossec.log" -Tail 20   # 看日志
Restart-Service WazuhSvc                                        # 重启
Start-Service WazuhSvc                                          # 启动
Stop-Service WazuhSvc                                           # 停止
Test-NetConnection <服务器IP> -Port 1514                        # 测试到服务器端口是否通
```

### Windows 卸载（如需）

```powershell
# 控制面板 → 程序和功能 → 卸载 "Wazuh Agent"；或命令：
Get-Package -Name "*Wazuh*" | Uninstall-Package
```

```bash
cd ~/Downloads
```

```bash
sudo ./install-macos.command <服务器IP> "设备名" "注册密码"
```

```bash
sudo /Library/Ossec/bin/wazuh-control status                   # 各进程 running

sudo /Library/Ossec/bin/wazuh-control restart                  # 重启
sudo /Library/Ossec/bin/wazuh-control start                    # 启动
sudo /Library/Ossec/bin/wazuh-control stop                     # 停止
nc -vz <服务器IP> 1514                                          # 测试到服务器端口是否通
```

```bash
sudo /Library/Ossec/bin/wazuh-control stop
sudo /bin/rm -rf /Library/Ossec
sudo /bin/launchctl remove com.wazuh.agent 2>/dev/null || true
```

## C2. Linux 电脑/服务器装 Agent

> 自动适配 Ubuntu/Debian/CentOS/RHEL/SUSE，装与服务器一致的版本。

### 方式①：一条命令在线安装（推荐）

## 5. GuizangAI对接说明.md

# GuizangAI 对接说明

> 这份文档用于部署或联调 GuizangAI 推理服务。
> 系统其余部分（Wazuh 采集、聚合、调度、数据库、仪表盘）已内置对接能力。

## 1. 只改配置即可

当前保留 `GUIZANGAI_*` 环境变量名作为兼容配置项；对外产品名称统一显示为 **GuizangAI**。

最小配置（`server/.env`）：

```ini
GUIZANGAI_BASE_URL=http://<GuizangAI机器IP>:1234
GUIZANGAI_API_STYLE=openai
GUIZANGAI_MODEL=guizangai-soc100-1.5b:q4
# 需要鉴权再填 GUIZANGAI_API_KEY；路径特殊再填 GUIZANGAI_CHAT_PATH
```

Ollama 原生接口的推荐配置：

```ini
GUIZANGAI_BASE_URL=http://host.docker.internal:11434
GUIZANGAI_API_STYLE=ollama
GUIZANGAI_MODEL=guizangai-soc100-1.5b:q4
GUIZANGAI_NUM_CTX=4096
GUIZANGAI_KEEP_ALIVE=30m
GUIZANGAI_NUM_GPU=999
GUIZANGAI_MAX_NEW_TOKENS=96
GUIZANGAI_TEMPERATURE=0.4
```

小机器可先降低资源占用：

```ini
GUIZANGAI_NUM_CTX=2048
GUIZANGAI_NUM_GPU=0
GUIZANGAI_ANALYSIS_CONCURRENCY=1
```

支持的接口风格：

| 配置值 | 适用后端 | 默认路径 |
| --- | --- | --- |
| `openai` | OpenAI 兼容服务 | `/v1/chat/completions` |
| `ollama` | Ollama 原生 | `/api/chat` |
| `custom_sse` | 现有 `/chat` + SSE 流式后端 | `/chat` |

对接完成后，仪表盘上 AI 卡片会显示 `GuizangAI 已接入`。

## 2. 联调

```bash
docker compose restart bff
curl -X POST http://<服务器IP>:8080/api/ai/run | jq .
```

如果返回里没有 `_error`，说明 GuizangAI 调用和 JSON 解析正常；如果失败，系统会自动回退规则/Mock 结果，仪表盘不会空白。

查看性能基线：

```bash
curl http://<服务器IP>:8080/api/ai/status | jq .
```

关注字段：

- `tokens_per_second` / `avg_tokens_per_second`：推理吞吐。
- `latest_seconds`：最近一次模型调用耗时。
- `recent_error_count`：最近分析结果里的模型或解析失败次数。
- `recent_cached_count`：最近分析结果里的缓存复用次数。

## 3. 效率参数

系统会在摘要没有变化时复用上一轮 AI 结论，避免周期任务重复推理。相关配置：

```ini
GUIZANGAI_ANALYSIS_CACHE_TTL_SECONDS=600
GUIZANGAI_ADVICE_CACHE_TTL_SECONDS=3600
GUIZANGAI_ANALYSIS_CONCURRENCY=1
ANALYSIS_INTERVAL_MINUTES=1
```

建议：

- 默认保持 `GUIZANGAI_ANALYSIS_CONCURRENCY=1`，最稳；确认 Ollama 单机并发能力后再调到 2 或 3。
- `ANALYSIS_INTERVAL_MINUTES` 生产建议 1-5 分钟；摘要缓存会跳过无变化数据。
- 开启原始日志时先降低 `GUIZANGAI_RAW_LOGS_MAX`，再根据模型上下文逐步增加。

## 4. 迁移清单

迁移到另一台机器时按这个顺序处理：

1. 在目标机安装 Ollama 或 OpenAI 兼容推理服务。
2. 将已解压的 `models/guizangAI/` 或 `guizangAI.gz` 放到项目根目录，运行 `./server/deploy.sh`；脚本会优先注册 Ollama 模型 `guizangai-soc100-1.5b:q4`，压缩包解压成功后会自动删除。
3. 如果没有本地模型包，可临时回退到公开模型：`ollama pull qwen2.5:3b`，并把 `GUIZANGAI_MODEL` 改成 `qwen2.5:3b`。
4. 在 `server/.env` 修改 `GUIZANGAI_BASE_URL`、`GUIZANGAI_API_STYLE`、`GUIZANGAI_MODEL`。
5. Linux Docker 场景可继续使用 `http://host.docker.internal:11434`；`docker-compose.yml` 已包含 `host-gateway` 映射。
6. 重启 BFF：`docker compose restart bff`。
7. 用 `/api/ai/status` 和 `/api/ai/run` 验证连通性、耗时、错误数。
8. 如需临时回滚，清空 `GUIZANGAI_BASE_URL`，系统会进入 Mock 模式。

## 5. 后续真正微调

当前仓库没有训练数据和权重微调脚本，因此不建议直接做 LoRA/QLoRA。更稳妥的流程：

1. 导出真实告警摘要、原始上下文、AI 输出和人工修正结果。
2. 整理成 JSONL，每条包含 `instruction`、`input`、`expected_output`。
3. 脱敏设备名、IP、路径、用户名等客户敏感信息。
4. 用单独训练环境做 LoRA/QLoRA，再量化成 Ollama 可加载格式。
5. 用 Modelfile 固化推理参数，迁移时同步模型标签、权重文件和 `.env`。

- GuizangAI 必须返回 JSON；内置提示词已强约束。
- 原始日志默认不发送，避免上下文过大；如需开启，用 `GUIZANGAI_SEND_RAW_LOGS`、`GUIZANGAI_RAW_LOGS_MAX`、`GUIZANGAI_RAW_LOGS_FIELDS` 控制。

## 6. 模型对接说明.md

# AI 推理服务对接说明（交给部署模型那台机器的 Cursor）

> 这份文档是给**部署 AI 推理服务的那台机器上的 Cursor**看的。
> 本系统其余部分（采集引擎、聚合、调度、数据库、仪表盘）已经全部跑通，
> 未接入模型时以 **Mock 模式**输出分析结论。你只需要让 BFF 能真正调用 AI 模型即可。
> （当前默认对接的是本地 Ollama 上的 `qwen2.5:7b`，换任意其它模型同理。）

## 1. 大多数情况：只改 `.env`，不用改代码

接入点已内置 **3 种主流接口风格**，通过环境变量切换（`server/.env`）：

| AI_API_STYLE | 适用后端 | 默认路径 |
| --- | --- | --- |
| `openai`（默认） | LM Studio / vLLM / llama.cpp / Ollama(/v1) 等 OpenAI 兼容服务 | `/v1/chat/completions` |
| `ollama` | Ollama 原生 | `/api/chat` |
| `custom_sse` | 现有 `/chat` + SSE 流式后端 | `/chat` |

最小配置（以本地 Ollama 为例）：

```ini
AI_BASE_URL=http://host.docker.internal:11434
AI_API_STYLE=ollama
AI_MODEL=qwen2.5:7b
# 需要鉴权再填 AI_API_KEY；路径特殊再填 AI_CHAT_PATH
```

对接完成的标志：仪表盘上 AI 卡片的来源徽标从 `Mock` 变成 `AI`。

> 仅当你的接口是上述 3 种都不符合的私有协议时，才需要动代码——见第 3 节。
> 代码里唯一的耦合点是 `bff/app/ai_client.py`：在 `_call()` 分发处加一个 `_call_xxx()` 即可，其余全不用动。

## 2. 调用契约（输入/输出）

BFF 会对**每个预设任务**调用一次 `analyze(task_key, summary, raw_events)`：

- **输入**：①一个已经聚合好的 JSON 摘要(示例见第 5 节)；②该时间窗内的**原始告警日志全量明细**(JSON 数组，整条 `_source`)。两者拼进同一个提示词发给模型。
- **输出**：必须是**一个 JSON 对象**，字段与 `bff/app/prompts.py` 中该任务的 `schema` 对应。

> ⚠️ 原始日志可能很大：由 `server/.env` 的 `AI_SEND_RAW_LOGS`/`AI_RAW_LOGS_MAX`/`AI_RAW_LOGS_FIELDS` 控制是否发送、最多发多少条、只发哪些字段。请确保模型的上下文窗口(`num_ctx`/`max_model_len` 等)足够大，否则会被截断或超时——可调小 `AI_RAW_LOGS_MAX` 或裁字段。

当前预设三个任务（`PRESET_TASKS`）：

| task_key | 用途 | 期望返回字段 |
| --- | --- | --- |
| `overview` | 整体安全态势 | `risk_level, risk_score, headline, summary, top_actions[]` |
| `alert_triage` | 告警归并降噪 | `clusters[]{category,count,meaning,severity}` |
| `compliance` | 合规加固建议 | `worst_endpoint, worst_score, recommendations[]` |

> 客户**不会**和模型对话。这些任务由我方预设、后台每 10 分钟自动批量执行，结果直接填充仪表盘固定卡片。

## 3. 仅特殊私有接口才需改代码

3 种内置风格已覆盖绝大多数本地推理服务（见第 1 节），它们在 `ai_client.py` 里分别是
`_call_openai()` / `_call_ollama()` / `_call_custom_sse()`，由 `_call()` 按 `AI_API_STYLE` 分发。

如果你的模型服务是上述都不符合的私有协议，照葫芦画瓢加一个方法即可：

```python
# 1) 在 AiClient 里新增
async def _call_myapi(self, message: str) -> str:
    url = self.base_url + (settings.ai_chat_path or "/your/path")
    payload = { ... 按你的后端拼 ... }
    async with httpx.AsyncClient(verify=settings.verify_tls, timeout=settings.ai_timeout_seconds) as c:
        r = await c.post(url, json=payload)
        r.raise_for_status()
        return r.json()[".../到模型输出文本的路径"]

# 2) 在 _call() 分发处加一行
if style == "myapi":
    return await self._call_myapi(message)
```

然后 `.env` 里 `AI_API_STYLE=myapi`。`_call()` 只要返回**模型输出的纯文本**，
后续的 JSON 抽取（`_extract_json`）和落库都由现有代码完成，你不用管。

> 提示：若你的后端支持"强制 JSON 输出"（OpenAI 的 `response_format={"type":"json_object"}`、
> Ollama 的 `format=json`），开启后稳定性最好。内置的 `ollama` 风格已默认开 `format=json`。

## 4. 配置与联调

1. 在客户服务器 `server/.env` 填：`AI_BASE_URL=http://<模型服务IP>:<端口>`
2. 重启后端：`docker compose restart bff`
3. 联调单个任务（不依赖前端）：

## 7. 注意事项

- **必须只返回 JSON**：`prompts.py` 已在提示词里强约束；若你的模型仍爱加解释文字，优先用 `format=json`/`response_format` 之类的强制 JSON 选项。
- **原始日志全量发送**：当前已把时间窗内原始告警明细随摘要一起发给模型(由 `AI_SEND_RAW_LOGS` 等开关控制)。务必确保模型上下文窗口足够，必要时调小 `AI_RAW_LOGS_MAX` 或用 `AI_RAW_LOGS_FIELDS` 只取关键字段。
- **失败要安全降级**：`analyze()` 已做 try/except，调用失败会回退 mock，保证仪表盘永远不空。
- **隐私**：原始日志含设备名/IP、文件路径、进程/命令等明细，比摘要敏感得多。是否出网、发送范围需严格符合与客户的合同约定；如需脱敏可用 `AI_RAW_LOGS_FIELDS` 限定字段。
